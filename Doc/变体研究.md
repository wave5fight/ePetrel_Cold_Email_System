# AI Spintax 裂变与发件箱动态排重完整技术方案

本方案旨在为开源发信平台提供底层的风控和文本渲染引擎架构。用户只需在前端编写 **1 个** 包含自定义变量的原始模板，系统即可全自动、安全地完成变体裂变与发送。

## 一、 系统架构设计与数据流向

整个文本处理与发信流控分为四个阶段：**变量隔离 $\rightarrow$ AI 语义裂变 $\rightarrow$ 变量还原 $\rightarrow$ 种子指纹去重渲染**。

```
[用户输入: 原始单模版] 
         │ (含有 {{firstName}}, {{companyName}} 等自定义属性)
         ▼ 
 1. 变量隔离层 (Tokenization) ───> 用正则表达式将变量锁死替换为特殊字符串标识
         │
         ▼ (将锁死变量后的文本连同“行业/语气” Context 喂给 LLM API)
 2. AI 语义裂变层 (LLM Engine) ───> 自动切片并输出标准 {A|B|C} 嵌套语法
         │
         ▼ 
 3. 变量还原层 (Detokenization) ──> 将特殊字符串标识恢复为标准大括号格式，存入任务库
         │
         ▼ 
 4. 队列去重渲染层 (Queue Engine) ─> 基于 sender_id + sequence_no 唯一种子渲染，MD5指纹校验

```

## 二、 核心技术模块深度拆解

### 1. 变量隔离与安全 AI 裂变（解决变量不被改坏、适配不同行业）

为了防止 AI 在改写和生成变体时自作聪明地修改或翻译 `{{firstName}}`、`{{City}}` 等用户绑定变量，系统必须在调用大模型前进行“数据脱敏加锁”。

同时，为了适应不同行业的开源用户，前端应提供 **“行业类型 (Industry)”** 和 **“话术风格 (Tone)”** 的下拉菜单作为 AI 的上下文输入。

**后台核心处理代码 (Python 示例)：**

```python
import re

def generate_safe_ai_spintax(user_template, industry, tone, ai_client):
    """
    第一步：通过正则锁定并保护用户绑定的所有变量（支持无限个自定义属性值）
    """
    # 匹配用户所有的双大括号变量，如 {{firstName}}, {{any_other_property}}
    variable_pattern = re.compile(r'\{\{([^}]+)\}\}')
    
    token_map = {}
    
    def replacer(match):
        var_name = match.group(1).strip()
        # 生成一个 AI 绝对不会去胡乱修改或翻译的特殊大写字符串标识
        token = f"__PROTECTED_VAR_{len(token_map)}__"
        token_map[token] = f"{{{{{var_name}}}}}" # 记录原貌，如 {{firstName}}
        return token

    # 将用户原文中的 {{变量}} 变成安全 Token
    locked_text = variable_pattern.sub(replacer, user_template)
    
    """
    第二步：封装高级结构化 Prompt，调用 AI 引擎（如 GPT-4o-mini 或 DeepSeek）
    """
    ai_prompt = f"""
    # Role
    你是一个顶级 B2B 冷发信风控专家，精通各大邮件服务商（Google/Outlook）的文本相似度过滤算法。

    # Task
    请将用户输入的原始邮件文本，改写为极其自然的、高密度的 Spintax 格式（使用 `{{选项A|选项B|选项C}}` 语法）。

    # Context
    当前用户属于 【{industry}】 行业，希望用 【{tone}】 的语气和口吻来进行改写，使其高度符合北美本地人的商务/口语习惯。

    # Constraints
    1. 绝对不能修改、删除或翻译任何带有双下划线的保护标记（例如 __PROTECTED_VAR_0__）。它们必须完好地保留在改写后的上下文里。
    2. 每句话必须生成至少 3 个在语法结构、同义词、表达方式上完全不同的变体。
    3. 招呼语、第一句破冰、核心痛点、行动呼吁（CTA）必须全部做 Spintax 切片。
    4. 严格只输出包含 Spintax 的邮件正文，不要带有任何解释、Markdown 代码块（如 ```text）或问候语。

    # User Input Text
    {locked_text}
    """
    
    # 调用 AI（此处为伪代码，可适配各类 LLM SDK）
    ai_response = ai_client.call(prompt=ai_prompt) 
    
    """
    第三步：反向映射，将 Token 还原为系统标准变量格式，返回结果存入数据库
    """
    final_spintax_template = ai_response
    for token, original_variable in token_map.items():
        final_spintax_template = final_spintax_template.replace(token, original_variable)
        
    return final_spintax_template

```

**输入输出流演示：**

* **用户在前端输入：** *"Hi {{firstName}}, I saw your shop on Google Maps. We can help your bakery in {{City}}."*
* **AI 在后台返回（并已还原）：** *`"{Hey|Hi|Hello} {{firstName}}, {Quick question—|Just saw your spot on Maps: } we help shops in {{City}} {get clients|boost sales}."`*

### 2. 队列渲染与“同账号绝对排重”算法（解决内容撞车）

当任务进入发送队列（如使用 Celery, BullMQ 或 Redis Queue）后，调度引擎绝对不能使用完全随机的数字来解开 Spintax，否则在并发或连续发信时，由于概率碰撞，同一个发件箱仍会发出相同文本。

**实现逻辑：**
系统必须将 **`当前发件箱的唯一数据库 ID (sender_id)`** 与 **`该发件箱在该任务中的发送累加序号 (sequence_no)`** 拼接，作为伪随机数生成器（PRNG）的**种子（Seed）**。

**核心动态去重渲染代码：**

```python
import re
import random
import hashlib

def render_unique_email(spintax_text, sender_id, sequence_no, lead_data, redis_client):
    """
    基于发件箱状态种子，动态渲染一封绝对去重的邮件，并进行哈希指纹校验
    """
    max_retries = 5
    current_retry = 0
    
    while current_retry < max_retries:
        # 1. 引入唯一种子。随着 sequence_no 的递增，同一个发件箱连续生成的路径绝不相同
        # 如果遭遇指纹重复，叠加重试次数（current_retry）强行让种子发生偏移
        unique_seed = f"{sender_id}_{sequence_no}_{current_retry}"
        random.seed(unique_seed)
        
        # 2. 递归解析并随机选择 Spintax 块 {A|B|C}
        spintax_pattern = re.compile(r'\{([^{}]+)\}')
        current_text = spintax_text
        
        while spintax_pattern.search(current_text):
            def replace_match(match):
                options = match.group(1).split('|')
                return random.choice(options) # 此处的选择受当前 unique_seed 约束
            current_text = spintax_pattern.sub(replace_match, current_text)
            
        # 3. 计算文本哈希指纹（Fingerprint），死保同一个发件箱在 24 小时内不发重复文本
        text_hash = hashlib.md5(current_text.encode('utf-8')).hexdigest()
        redis_key = f"fingerprint:{sender_id}:{text_hash}"
        
        # 检查 Redis 中是否存在该指纹
        if not redis_client.exists(redis_key):
            # 指纹通过，锁死该指纹 24 小时，防止撞车
            redis_client.setex(redis_key, 86400, "1")
            
            # 4. 替换动态变量（注入真正的客户属性数据，如名字、公司名）
            for key, value in lead_data.items():
                placeholder = f"{{{{{key}}}}}"
                current_text = current_text.replace(placeholder, str(value))
                
            return current_text # 返回最终发信文本
            
        else:
            # 遭遇指纹撞车，触发重试，更换种子重新投掷随机数
            current_retry += 1
            
    # 超过最大重试次数依然撞车（通常因为用户的 Spintax 密度太低，组合数耗尽），降级直接返回
    return current_text

```

## 三、 开源 APP 调度引擎的三条铁律（研发红线）

为了让你的开源发信 APP 在风控表现上超越同行，你的底层发信流控（Scheduler）必须强制实现以下三条规则：

1. **多发件箱轮询策略 (Round-Robin)：** 队列分发任务时，严禁让“发件箱 A”连续发完 50 封再换“发件箱 B”。必须是：发件箱 A 发给 Lead 1 $\rightarrow$ 发件箱 B 发给 Lead 2 $\rightarrow$ 发件箱 C 发给 Lead 3，在时序上最大化拉开单个账号的发信间隔。
2. **强制随机时间抖动 (Time Jitter)：** 用户设置了“每隔 60 秒发一封信”，系统绝不能死板地在第 60.000 秒触发 SMTP。必须在底层加入 `± random.randint(5, 25)` 秒的随机抖动，让发信行为模拟真人在客户端点击发送的自然节奏。
3. **退弹率熔断机制 (Bounce Rate Fuse)：** 监控每个发件箱的退信报错。一旦某个发件箱连续出现 3 次“Address not found”或被服务商拒绝（550 Error），系统必须**立刻自动暂停该发件箱的队列**，防止因为死信过多导致整个发信域名的信誉度崩盘。
